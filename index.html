    <h1>Goals vs Projects vs Tasks</h1>

    <div class="subtitle">

      A reference guide to help George classify ideas quickly, reduce mental clutter, and decide what deserves strategic attention.

    </div>


    <div class="callout">

      Goal = the business result. Project = the build that moves the result forward. Task = the next action inside the build.

    </div>


    <h2>1. Why This Matters</h2>

    <ul>

      <li>George has many ideas, notes, and priorities competing for attention.</li>

      <li>The issue is not lack of ideas. The issue is classification.</li>

      <li>When goals, projects, and tasks are mixed together, everything feels equally important.</li>

      <li>The purpose of this guide is to help George sort work faster and decide what belongs at what level.</li>

    </ul>


    <div class="warning">

      <strong>Strategic rule:</strong> George should not spend $10k/hour energy sorting low-level tasks. The team should help convert raw ideas into clean goals, projects, and tasks.

    </div>


    <h2>2. Core Definitions</h2>


    <table>

      <thead>

        <tr>

          <th>Category</th>

      </thead>

        <tr>

          <td>What needs to be built, fixed, launched, or completed?</td>

    <h2>4. Fast Decision Rules</h2>


    <div class="decision-box">

      <h3>Question 1: Is this an outcome or a deliverable?</h3>

      <ul>

        <li><strong>Outcome</strong> = Goal</li>

        <li><strong>Deliverable</strong> = Project</li>

      </ul>

    </div>


    <div class="decision-box">

      <h3>Question 2: Does this require sequencing?</h3>

      <ul>

        <li><strong>Yes</strong> = Project</li>

        <li><strong>No</strong> = Task</li>

      </ul>

    </div>


    <div class="decision-box">

      <h3>Question 3: Does this deserve George’s strategic attention?</h3>

      <ul>

        <li><strong>Yes</strong> = Goal or Quarterly Project</li>

        <li><strong>No</strong> = Mini Project, One-off Project, or Task</li>

      </ul>

    </div>

      The current project system already classifies projects as Quarterly, Mini, One-off, and Task.

      This guide adds the missing layer above the project framework: the difference between a goal and a project.

    <h2>7. George-Friendly Language</h2>


    <div class="simple-line">

      Do not ask: “Is this a goal or project?”

    </div>


    <p>That question creates overthinking.</p>


    <div class="simple-line">

      Ask instead: “Is this the result, the build, or the next action?”

    </div>


    <table>

      <thead>

        <tr>

          <th>If It Is...</th>

          <th>Then It Is...</th>

        </tr>

      </thead>

      <tbody>

        <tr>

          <td>The result we want</td>

          <td>Goal</td>

        </tr>

        <tr>

          <td>The thing we need to build or complete</td>

          <td>Project</td>

        </tr>

        <tr>

          <td>The next action to move it forward</td>

          <td>Task</td>

        </tr>

      </tbody>

    </table>


    <h2>8. Recommended Operating Rule</h2>


    <ul>

      <li>Goals should be few.</li>

      <li>Projects can be many.</li>

      <li>Tasks should move fast and be delegated.</li>

      <li>Not every important project should become a goal.</li>

      <li>Not every task deserves George’s review.</li>

      <li>When George brain-dumps, the team should sort the list for him.</li>

    </ul>


    <div class="warning">

      <strong>Management principle:</strong> George should provide raw direction. The team should convert it into clean goals, projects, and tasks before asking for final confirmation.

    </div>


    <h2>9. Final Working Definition</h2>


    <table>

      <thead>

        <tr>

          <th>Term</th>

          <th>Final Definition</th>

        </tr>

      </thead>

      <tbody>

        <tr>

          <td>Goal</td>

        </tr>

        <tr>

          <td>Project</td>

          <td>A temporary, multi-step body of work that creates a defined deliverable in service of a goal.</td>

        </tr>

        <tr>

          <td>Task</td>

          <td>A single action with no meaningful sequencing. It supports a project or operating process.</td>
